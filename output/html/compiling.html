Der Zugriff auf Samba Source Code via CVS

Einführung

Samba wird in einer offenen Umgebung entwickelt. Die Entwickler benutzen das Concurrent Versioning System (CVS), um mittels “checkin”, (auch als “commit” bekannt), neuen Quelltext einzufügen. Auf die verschiedenen CVS-Zweige von Samba kann mittels anonymem CVS zugegriffen werden, dies unter Verwendung der in diesem Kapitel enthaltenen Anleitungen.

Dieses Kapitel ist eine geänderte Fassung der Instruktionen, die man unter http://samba.org/samba/cvs.html findet.

CVS-Zugriff auf samba.org

Die Maschine samba.org führt ein offen zugängliches CVS-repository. Damit ermöglicht sie den Zugriff auf den Quelltext mehrerer Packages, einschließlich Samba, rsync, distcc, ccache, und jitterbug. Es gibt zwei hauptsächliche Arten, um auf den CVS-server auf diesem host zuzugreifen:

Zugriff via CVSweb

Sie können so unter Verwendung Ihres bevorzugten WWW-Browsers auf den QUelltext zugreifen. Dies erlaubt den Zugriff auf einzelne Dateien im Repository genauso wie auf den Verlauf der Revisionen und die Commit-logs der einzelnen Dateien. Man kann damit auch diff-listings zwischen zwei Versionen erstellen.

Verwenden Sie die URL: http://samba.org/cgi-bin/CVSweb

Zugriff via CVS

Sie können auf den Quelltext auch mittels eines normalen CVS-clients zugreifen. Dies gibt Ihnen mehr Kontrolle darüber, was Sie mit dem Repository tun können, und erlaubt Ihnen, ganze Quelltext-Bäume mittels checkout abzuholen, und sie mittels normaler CVS-Befehle am Stand zu halten. Dies ist die bevorzugte Variante des Zugriffs, wenn Sie ein Entwickler sind, und nicht nur ein gelegentlicher Besucher.

Um den letzten CVS-Quelltext herunterzuladen, geben Sie folgende URL in Ihren Browser ein: http://www.cyclic.com/. und klicken SIe auf den Link “How to get CVS”. CVS ist freie Software unter der GNU GPL (so wie auch Samba). Wir machen Sie darauf aufmerksam, daß es auch einige graphische CVS-clients gibt, die eine graphische Oberfläche für die manchmal sehr nüchternen CVS-Befehle anbieten. Links zu diesen clients sind auch auf der Cyclic Web site verfügbar.

Um Zugriff via anonymem CVS zu erlangen, führen Sie die folgenden Schritte durch. In diesem Beispiel wird angenommen, daß Sie eine Kopie des Samba-Quelltexts haben wollen. Für die anderen Quelltext-Repositories auf diesem System ersetzen sie einfach den Package-Namen.

Procedure 36.1. Samba über CVS beziehen

  1. Installieren Sie eine aktuelle Version von CVS. Alles was Sie brauchen, ist eine Kopie des CVS-client-Programmes.

  2. Führen Sie folgenden Befehl aus:

    cvs -d :pserver:cvs@samba.org:/cvsroot login

  3. Wenn Sie nach dem Passwort gefragt werden, tippen Sie cvs.

  4. Führen Sie folgenden Befehl aus:

    cvs -d :pserver:CVS@samba.org:/cvsroot co samba.

    Dies wird ein Verzeichnis namens samba anlegen, das den neuesten Samba-Quelltext (den CVS-Zweig namens HEAD) enthält. Dies entspricht momentan dem Entwicklungs-Code für Samba 3.0.

    CVS-Zweige, die nicht HEAD sind, können bezogen werden, indem man die Option -r mit einem tag-Namen verwendet. Eine Liste von tag-Namen kann auf der Seite “Development” der Samba-Web-Site gefunden werden. Eine gebräuchliche Anfrage ist die nach dem letzten 3.0 Release-Code. Dies könnte wie folgt geschehen:

    cvs -d :pserver:cvs@samba.org:/cvsroot co -r SAMBA_3_0 samba.

  5. Wann immer Sie die letzten Code-Änderungen übernehmen wollen, verwenden Sie folgenden Befehl innerhalb des Samba-Verzeichnisses:

    cvs update -d -P

Zugriff auf die Samba-Quelltexte via rsync und ftp

pserver.samba.org exportiert auch ungepackte Kopien der meisten Teile des CVS-Baums unter ftp://pserver.samba.org/pub/unpacked und auch via anonymem rsync unter rsync://pserver.samba.org/ftp/unpacked/. Wir empfehlen, rsync anstatt ftp zu verwenden. Sehen Sie the rsync homepage für mehr Informationen zu rsync.

Der Nachteil der ungepackten Bäume ist, daß sie kein automatisches "Merge" von lokalen Änderungen zulassen, wie es CVS tut. rsync-Zugriff ist das praktischste Verfahren bei einer frischen Installation.

Überprüfen der PGP-Signatur von Samba

Es wird wärmstens empfohlen, daß Sie die PGP-Signatur jeder Quelltext-Datei vor der Installation überprüfen. Sogar wenn Sie die Datei nicht von einem Mirror-Server laden, sollte das Prüfen der PGP-Signatur zum Reflex werden. Viele nutzen heute das GNU-GPG-Package anstatt von PGP. GPG kann anstatt PGP eingesetzt werden.

Nachdem dies gesagt wurde, laden Sie bitte folgende Dateien aus dem Netz: 

$ wget http://us1.samba.org/samba/ftp/samba-2.2.8a.tar.asc
$ wget http://us1.samba.org/samba/ftp/samba-pubkey.asc

Die erste Datei ist die PGP-Signatur für die Samba-Quelltext-Datei; die andere ist der öffentliche Samba-PGP-Schlüssel selbst. Importieren Sie den öffentlichen PGP-Schlüssel mittels: 

$ gpg --import samba-pubkey.asc

und überprüfen Sie die Integrität des Samba-Quelltexts mittels: 

$ gzip -d samba-2.2.8a.tar.gz
$ gpg --verify samba-2.2.8a.tar.asc

Wenn Sie eine Meldung wie “Good signature from Samba Distribution Verification Key...” erhalten, ist alles OK. Die Warnungen bezüglich “trust relationships” können ignoriert werden. Ein Beispiel für etwas, das Sie nicht sehen wollen: 

     gpg: BAD signature from “Samba Distribution Verification Key”

Kompilieren der Binärdateien

Um die Binärdateien mittels "build" zu erhalten, führen Sie zuerst das Programm ./configure im Quellverzeichnis aus. Dies sollte Samba automatisch für Ihr Betriebssystem konfigurieren. Sollten Sie darüber hinausgehende Wünsche haben, dann können Sie folgendes tun:

root# ./configure --help

So sehen Sie, welche speziellen Optionen aktiviert werden können. Nun können Sie ./configure mit jeglichen gewünschten Agrumenten ausführen:

root# ./configure [... arguments ...]

Das Ausführen von

root# make

wird die Binärdateien erstellen. Sobald diese erfolgreich kompiliert wurden, können Sie

root# make install

verwenden, um die Binärdateien (=Programme) und Man-pages zu installieren. Sie können Programme und Man-pages separat installieren:

root# make installbin

und

root# make installman

Bitte beachten Sie, daß Sie beim Upgrade von einer vorigen Version möglicherweise wissen wollen, daß dabei die alten Programme umbenannt werden, und zwar durch das Anhängen von “.old”. Sie können dann zur alten Version zurückkehren,

root# make revert

wenn Sie die neue Version als Desaster empfinden!

Das Kompilieren von Samba mit Active Directory Support

Um Samba mit ADS-Support zu kompilieren, müssen Sie folgendes auf Ihrem System installiert haben:

  • Die MIT- oder Heimdal-Kerberos development libraries (entweder selbst kompiliert oder aus einem Package).

  • Die OpenLDAP development libraries.

Wenn Ihre Kerberos libraries an einem vom Standard-Verzeichnis abweichenden Ort liegen, fügen Sie die configure option --with-krb5=DIR hinzu.

Nachdem Sie configure ausgeführt haben, prüfen Sie, ob die entstandene Datei include/config.h Zeilen wie diese enthält: 

#define HAVE_KRB5 1
#define HAVE_LDAP 1

Falls dem nicht so ist, hat configure Ihre KRB5 libraries oder LDAP libraries nicht gefunden. Konsultieren Sie config.log, um den Grund dafür zu finden und zu beheben.

Die Installation der erforderlichen Packages für Debian

Unter Debian müssen Sie folgende Packages installieren:

  • libkrb5-dev
  • krb5-user

>Die Installation der erforderlichen Packages für Red Hat Linux

Auf Red Hat Linux bedeutet das, daß Sie zumindest das brauchen:

  • krb5-workstation (für kinit)
  • krb5-libs (für das Linken)
  • krb5-devel (weil Sie aus den Quellen kompilieren)

(dies zusätzlich zur Standard-Entwicklungs-Umgebung.)

Wenn diese Dateien nicht auf Ihrem System installiert sind, sollten Sie die Installations-CDs überprüfen, um herauszufinden, wo diese Packages zu finden sind, und sie mit einem Werkzeug Ihrer Wahl installieren. Wenn Sie unsicher sind, welches Werkzeug zu verwenden ist, konsultieren Sie bitte die Red Hat Linux Dokumentation.

SuSE Linux Package Erfordernisse

SuSE Linux installiert Heimdal-Packages, die erforderlich sein könnten, um Binary-packages erstellen zu können. Sie sollten überprüfen, ob diese development-libraries auf Ihrem System installiert wurden.

SuSE Linux Samba RPMs unterstützen Kerberos. Bitte beziehen Sie sich auf die Dokumentation Ihres SuSE Linux Systems für Informationen zur SuSE-spezifischen Konfiguration. Weiters ist SuSE sehr aktiv in der Pflege von Samba-Packages, die alle verfügbaren Fähigkeiten von Samba bereitstellen. Sie sollten die Verwendung von SuSE-Samba-Packages in Erwägung ziehen, wo dies möglich ist.

Starten von smbd und nmbd

Sie müssen sich entscheiden, smbd und nmbd entweder als "daemons" oder mittels inetd zu starten. Versuchen Sie nicht, beides zu tun! Sie können sie entweder in inetd.conf eintragen und bei Bedarf von inetd oder xinetd starten lassen, oder sie als daemons entweder von der Befehlszeile oder in /etc/rc.local starten. Lesen Sie die Man-pages für Details zu den Befehls-Optionen. Lesen Sie im Besonderen den Abschnitt über den Benutzer, den Sie brauchen, um Samba zu starten. In vielen Fällen müssen Sie dazu root sein.

Der hauptsächliche Vorteil des Startens von smbd und nmbd mittels der empfohlenen daemon-Methode ist, daß sie dann etwas schneller auf die erste Verbindungs-Anfrage reagieren werden.

Starten aus der inetd.conf

Note

Das Folgende ist unterschiedlich, je nachdem, ob Sie NIS, NIS+ oder LDAP zur Verteilung Ihrer Dienst-Listen verwenden.

Sehen Sie sich /etc/services an. Was ist für port 139/tcp definiert? Wenn nichts definiert ist, fügen Sie eine Zeile wie diese hinzu:

netbios-ssn     139/tcp

Ähnlich für 137/udp, Sie sollten einen Eintrag haben wie:

netbios-ns	137/udp

Als nächstes editieren Sie Ihre /etc/inetd.conf und fügen zwei Zeilen wie diese hinzu:

		netbios-ssn stream tcp nowait root /usr/local/samba/bin/smbd smbd 
		netbios-ns dgram udp wait root /usr/local/samba/bin/nmbd nmbd

Die exakte Syntax von /etc/inetd.conf variiert zwischen UNIX's. Nehmen Sie sich andere Einträge in inetd.conf zur Vorlage.

Manche Distributionen verwenden xinetd anstatt von inetd. Konsultieren Sie das xinetd-Manual für Informationen zur Konfiguration.

Note

Manche UNIX's haben bereits Einträge wie netbios_ns (beachten Sie den Unterstrich) in /etc/services. Sie müssen /etc/services oder /etc/inetd.conf editieren, um beide konsistent zu erhalten.

Note

Auf vielen Systemen müssen Sie eventuell die Option interfaces in smb.conf verwenden, um die IP-Adresse und Netzmaske Ihrer Netz-Interfaces anzugeben. Führen Sie ifconfig als root aus, wenn Sie die Broadcast-Adresse Ihres Netzes nicht kennen. nmbd versucht diese bei seiner Ausführung zu bestimmen, scheitert aber auf manchen UNIX's.

Warning

Viele UNIX's akzeptieren nur ungefähr fünf Parameter auf der Befehlszeile in inetd.conf. Das bedeutet, daß Sie keine Leerzeichen zwischen Optionen und Argumenten verwenden sollten, bzw. daß Sie ein Skript verwenden und dies mittels inetd starten sollten.

Starten Sie inetd neu, bzw. senden Sie das Signal HUP.

			root# killall -HUP inetd

Alternative: smbd als Daemon starten

Um den Server als daemon zu starten, sollten Sie ein Skript wie das Folgende anlegen, zum Beispiel mit dem Namen startsmb.

		#!/bin/sh
		/usr/local/samba/bin/smbd -D 
		/usr/local/samba/bin/nmbd -D

Machen sie es ausführbar mittels chmod +x startsmb

Dann können startsmb von Hand starten oder in der Datei /etc/rc.local.

Um die Daemonen zu beenden, senden Sie ein KILL-Signal and die Prozesse nmbd and smbd.

Note

Wenn Sie das init-System in der Art von SVR4 verwenden, wollen Sie vielleicht das Beispiel-Skript examples/svr4-startup lesen, um Samba an Ihr System anzupassen.
