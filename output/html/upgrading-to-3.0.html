Dieses Kapitel befaßt sich ausschließlich mit den Unterschieden zwischen Samba-3.0.0 und Samba-2.2.8a. Es zeigt, wo in der Konfiguration Parameter geändert wurden, und stellt eine einfache Anleitung für den Umstieg von 2.2.x auf 3.0.0 dar.

Kurzanleitung zur Migration

Das Standard-Verhalten von Samba-3.0.0 sollte ungefähr dasselbe sein wie das von Samba-2.2.x. Das Standard-Verhalten, wenn der neue Parameter passdb backend nicht in der Datei smb.conf gesetzt ist, entspricht dem selben Standard-Verhalten von Samba-2.2.x mit encrypt passwords = Yes, und benutzt die Datenbank smbpasswd.

Also warum dann sagen, das Verhalten sollte ungefähr gleich wie bei Samba-2.2.x sein? Weil Samba-3 neue Protokolle beherrscht, wie Unterstützung für Unicode, was darin resultieren kann, daß anderer Protokoll-Code verwendet werden könnte. Das neue Verhalten unter solchen Umständen ist nicht exakt dasselbe wie das vorherige. Die gute Nachricht dabei ist, daß die Domänen- und Maschinen-SIDs beim Upgrade beibehalten werden.

Wenn das Samba-2.2.x-System ein LDAP-Backend verwendet hat, und keine Zeit vorhanden ist, um die LDAP-Datenbank upzudaten, stellen Sie sicher, daß passdb backend = ldapsam_compat in der Datei smb.conf gesetzt ist. Das restliche Verhalten sollte mehr oder minder das gleiche bleiben. Zu einem späteren Zeitpunkt, wenn Zeit vorhanden ist, um ein neues zu Samba-3 kompatibles LDAP-Backend zu implementieren, ist es möglich, die alte LDAP-Datenbank unter Verwendung des Befehls pdbedit zu migrieren. Neue Features in Samba-3

Die wichtigsten neuen Features sind:

  1. Active Directory Support. Dieses Release kann sich einer ADS realm als Mitglieds-Server anschliessen und Benutzer mittels LDAP/kerberos authentifizieren.

  2. Unicode Support. Samba kann nun Unicode on-the-fly verhandeln und es gibt intern eine viel bessere Infrastruktur für multi-byte- und Unicode-Zeichensätze.

  3. Neues Authentifizierungs-System. Das interne Authentifizierungs-System wurde fast komplett neu geschrieben. Die meisten Veränderungen sind intern, aber auch das neue Authoring-System ist sehr vielfältig konfigurierbar.

  4. Neues Dateinamen-mangling-System. Das Dateinamen-mangling-System wurde komplett neu geschrieben. Eine interne Datenbank speichert nun die "mangling-maps".

  5. Neuer Befehl “net”. Der neue Befehl “net” wurde hinzugefügt. Er ist ähnlich dem Windows-Befehl “net”. Eventuell werden einige andere Werkzeuge (wie smbpasswd) durch Sub-Befehle in “net” ersetzt.

  6. Samba kann nun NT-style status32-codes on the wire verhandeln. Dies verbessert die Handhabung von Fehlern und deren Codes deutlich.

  7. Bessere Windows-200x/XP-Druck-Unterstützung, einschließlich der Bereitstellung von Drucker-Attributen im ADS.

  8. Neue ladbare RPC-Module für passdb-Backends und Zeichensätze.

  9. Neue standardmässige Dual-Dämon-Unterstützung für Winbind mit besserer Performance.

  10. Unterstützung für die Migration von einer Windows NT 4.0-Domäne auf eine Samba-Domäne unter Beibehaltung von Benutzer-, Gruppen- und Domänen-SIDs.

  11. Unterstützung für das Herstellen von Vertrauensstellungen mit Windows NT 4.0-Domänen-Controllern.

  12. Initial-Unterstützung für eine verteilte Winbind-Architektur, unter Verwendung eines LDAP-Verzeichnisses zur Speicherung der Zuordnungen von SIDs auf UIDs/GIDs.

  13. Grosse Updates im Samba Dokumentations-Baum.

  14. Volle Unterstützung für Client- und Server-Signing, um Kompatibilität mit den Standard-Sicherheits-Einstellungen von Windows 2003 zu sichern.

Und noch viele weitere Verbesserungen!

Änderungen im Verhalten

Die folgenden Themen snd bekannte Veränderungen im Verhalten zwischen Samba-2.2 und Samba-3, die manche Installationen beeinflussen können.

  1. Wenn Samba-2.2 als Mitglied einer Windows-Domäne arbeitet, würde es alle Benutzer, die vom entfernten DC authentifiziert wurden, dem “guest account” zuordnen, wenn keine UID mittels des Aufrufs getpwnam() erhalten werden kann. Samba-3 weist die Verbindung als NT_STATUS_LOGON_FAILURE ab. Es gibt derzeit keine Möglichkeit, das Verhalten von Samba-2.2 wiederherzustellen.

  2. Beim Hinzufügen von Maschinen zu einer Samba-2.2-Domäne wurde “add user script” verwendet, um die UNIX-Identität des Maschinen-Vertrauens-Kontos anzulegen. Samba-3 führt “add machine script” ein, das zu diesem Zweck spezifiziert werden muß. Samba-3 wird nicht auf die Verwendung von “add user script” zurückgreifen, wenn kein “add machine script” vorhanden ist.

Passdb-Backends und Authentifikation

Es gibt einige Änderungen, deren sich Samba-Administratoren beim Wechsel auf Samba-3 bewußt sein sollten.

  1. Verschlüsselte Passwörter wurden nun standardmässig aktiviert, um besser mit out-of-the-box Windows-Client-Installationen zusammenzuarbeiten. Das bedeutet, daß entweder (a) ein Samba-Konto für jeden Benutzer angelegt werden muß, oder daß (b) “encrypt passwords = no” explizit in smb.conf gesetzt sein muß.

  2. Die neue Option security = ads wurde inkludiert, um die Integration einer ADS-Domäne unter Verwendung der nativen Windows-Kerberos-5- und LDAP-Protokolle zu ermöglichen.

Samba-3 beinhaltet auch die Möglichkeit, Verkettungen von Authentifikations-Methoden (auth methods), und Konten-DB-Backends (passdb backend) einzusetzen. Bitte konsultieren Sie die manpage für smb.conf und ??? für Details. Obwohl beide Parameter auf vernünftige Standard-Werte gesetzt werden, ist es doch wahrscheinlich, daß Sie ihre Bedeutung kennenlernen wollen, um den korrekten Betrieb von Samba zu gewährleisten.

Bestimmte Funktionen des Befehls smbpasswd wurden auf das neue Werkzeug smbpasswd, das Werkzeug net und das neue pdbedit aufgeteilt. Details dazu in den jeweiligen manpages.

LDAP

Dieser Abschnitt umreißt die neuen Features betreffend Samba/LDAP-Integration.

Neues Schema

Eine neue Objektklasse (sambaSamAccount) wurde eingeführt, um die alte Klasse sambaAccount zu ersetzen. Diese Änderung hilft dabei, Attribute umzubenennen, um Kollisionen mit Attributen anderer Hersteller zu vermeiden. Es gibt ein Konverter-Skript (examples/LDAP/convertSambaAccount), das ein LDIF-file auf das neue Schema konvertiert.

Beispiel: 

		$ ldapsearch .... -b "ou=people,dc=..." > old.ldif
		$ convertSambaAccount <DOM SID> old.ldif new.ldif

Die <DOM SID> kann durch das Ausführen von 

$ net getlocalsid <DOMAINNAME>

auf dem Samba PDC (als root) abgefragt werden.

Das alte sambaAccount-Schema kann weiterhin verwendet werden, indem man das Passdb-Backend ldapsam_compat spezifiziert. Wie auch immer, sambaAccount und zugehörige Attribute wurden in den "historical"-Abschnitt der Schema-Datei verschoben und müssen erst auskommentiert werden, um sie verwenden zu können. Die Samba-2.2 Objektklassen-Deklaration für einen sambaAccount hat sich in der Samba-3 samba.schema-Datei nicht verändert.

Andere neue Objektklassen und deren Anwendungen beinhalten:

  • sambaDomain Domänen-Information zur Zuordnung von RIDs für Benutzer und Gruppen. Die Attribute werden im “ldap suffix”-Verzeichnis-Eintrag automatisch hinzugefügt, wenn ein UID/GID-Bereich für die idmap gesetzt wurde und das “ldapsam”-Passdb-Backend ausgewählt wurde.

  • sambaGroupMapping Ein Objekt, das die Beziehung zwischen einer posixGroup und einer Windows Gruppe/SID repräsentiert. Diese Einträge werden im “ldap group suffix” gespeichert und mittels des Befehls “net groupmap” verwaltet.

  • sambaUNIXIdPool Wird automatisch im “ldap idmap suffix” angelegt, und enthält die nächste verfügbare “idmap UID” und “idmap GID”.

  • sambaIdmapEntry Objekt, das eine Zuordnung zwischen SID und einer UNIX UID/GID speichert. Diese Objekte werden vom idmap_ldap-Modul je nach Bedarf angelegt.

Neuer Suffix für die Suche

Die folgenden neuen smb.conf-Parameter wurden hinzugefügt, um bestimmte LDAP-Abfragen zu unterstützen, wenn passdb backend = ldapsam://... spezifiziert wurde.

  • ldap suffix für die Suche nach Benutzer- und Maschinen-Konten.

  • ldap user suffix für das Speichern von Benutzer-Konten.

  • ldap machine suffix für das Speichern von Maschinen-Vertrauens-Konten.

  • ldap group suffix posixGroup/sambaGroupMapping-Einträge.

  • ldap idmap suffix sambaIdmapEntry-Objekte.

Wenn ein ldap suffix definiert ist, wird er an alle verbleibenden sub-suffix-Parameter angehängt. In diesem Fall ist die Reihenfolge der Suffixes in smb.conf wichtig. Plazieren Sie ldap suffix immer als ersten Eintrag in der Liste.

Wegen einer Beschränkung in Samba's Parsing von smb.conf sollten Sie die DNs nicht mit Anführungszeichen umschliessen.

IdMap LDAP Support

Samba-3 unterstützt ein LDAP-Backend für das idmap-Subsystem. Die folgenden Optionen informieren Samba, daß die idmap-Tabelle auf dem Verzeichnis-Server "onterose" gespeichert werden soll, in der Partition "ou=idmap,dc=quenya,dc=org".

[global]
...
idmap backend = ldap:ldap://onterose/
ldap idmap suffix = ou=idmap,dc=quenya,dc=org
idmap uid = 40000-50000
idmap gid = 40000-50000

Diese Konfiguration erlaubt es Winbind-Installationen auf mehreren Servern, sich einen UID/GID-Zahlenbereich zu teilen, und damit die Probleme in der Zusammenarbeit mit NFS zu vermeiden, die es in Samba-2.2 gab.
